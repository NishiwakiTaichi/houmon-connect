## 1. プロジェクト概要

### 1.1 アプリ名

**訪問コネクト** — 「訪問の予定をつなぐ。チームがつながる。」

勤務先の事業所名「つながる訪問看護リハビリステーション」の理念と響き合う名前として命名。
バラバラだった情報の流れ(Chatwork投稿・手帳・スプレッドシート・PDF)を1つに**つなぐ**こと、
看護とリハビリの**チームがつながる**ことを表現している。
(英語表記/リポジトリ名: `houmon-connect`)

### 1.2 背景・現状の課題
訪問看護ステーションでは、週間スケジュールを以下の流れで管理している。

```
スタッフがChatworkに変更を文章で投稿
　→ 管理者が手帳に手書きでメモ
　→ 毎週金曜にスプレッドシートへ転記
　→ PDF化してChatworkに投稿
```

この運用には次の問題がある。

| # | 課題 | 困っている人 |
|---|------|------------|
| 1 | 同じ情報を3〜4回転記しており、毎週多くの時間を要する | 管理者 |
| 2 | 報告の様式が統一されておらず、看護/リハの区別や内容が分かりにくい | 管理者 |
| 3 | PDF投稿後の変更が反映されず、当日のスケジュールが正しくないことがある | 全スタッフ |
| 4 | 報告の反映漏れ・調整中案件の報告漏れが起きる | 全員 |
| 5 | ケアマネへの連絡状況が把握できず、クレームに繋がることがある | 管理者 |
| 6 | 休止・入院時に「基本ルートで誰がいつ訪問しているか」を探すのが大変 | 管理者 |
| 7 | 新人同行の可否、男性/女性限定が紙の色塗りでしか分からない | 管理者 |
| 8 | スタッフの休み(半休・時間休)が共有されず、休み中に電話が来る | スタッフ |

### 1.3 アプリの目的
**「転記作業をゼロにし、常に最新のスケジュールを全員がどこからでも見られる状態にする」**

- スタッフ: 定型フォームで変更を登録するだけ。**登録した瞬間にスケジュールへ反映**される。訪問先からスマホで確認・登録できる
- 管理者: 転記・承認作業は不要。**変更ログと確認チェックで「見落としゼロ」を担保**する
- 全員: Webでいつでも最新のスケジュールを確認できる

### 1.4 データ管理の基本方針

| 方針 | 内容 |
|------|------|
| 全員で編集 | 利用者・基本ルート・変更は全スタッフが編集できる(現場が一番情報を持っているため) |
| 即時反映 | 変更は登録と同時にスケジュールへ反映される(承認待ちは存在しない) |
| 完全な記録 | 利用者・基本ルート・変更の登録/編集/取り消しはすべてログに残り、**消すことができない** |
| 軽量な監督 | 管理者は「確認チェック」を付けるだけ。反映を止める権限ではなく、内容を見たという印 |

> 「誰でも編集できる」と「すべて記録に残る」はセットの設計です。
> 権限で縛る代わりに、透明性(ログ)でデータの信頼性を担保します。

## 2. 用語定義

| 用語 | 意味 |
|------|------|
| 基本ルート | 「◯曜日の◯時に誰が誰を訪問する」という定期訪問の雛形 |
| 週間スケジュール | 基本ルートに変更・スタッフ休暇を反映した、特定の週の実際の予定 |
| 変更登録 | スタッフが行う「休み・振替・休止・再開」の登録。即時反映される |
| 変更ログ | 誰が・いつ・何を登録/編集/取り消したかの履歴。削除不可 |
| 確認チェック | 管理者が変更内容を確認した印。スケジュール反映の条件ではない |
| サービス区分 | 「看護」または「リハビリ」。ページ表示を分ける単位 |
| ケアマネ連絡状況 | 変更についてケアマネジャーへ連絡済みかどうか(未連絡/連絡済み) |

## 3. ユーザーと権限

| ロール | できること |
|--------|-----------|
| 一般スタッフ(staff) | スケジュール閲覧、変更の登録/編集/取り消し、**利用者の登録/編集**、**基本ルートの登録/編集**、自分の休暇登録、変更ログの閲覧 |
| 管理者(manager) | 上記すべて + 変更の確認チェック、スタッフアカウントの管理、他スタッフの休暇登録 |

- 職種(看護師/PT/OT/ST/事務)はユーザー情報として保持する(表示・絞り込みに使用)
- 看護管理者・リハビリ責任者は **共通の manager 権限** とする(MVP)
- 看護とリハビリは **ページ(タブ)を分けて表示** する

## 4. 機能要件

### 4.1 MVP(今回作る範囲)

#### F-1. 認証
- メールアドレス + パスワードでログイン/ログアウト(devise を使用)
- 新規スタッフの登録は管理者が行う(自由登録は不可)

#### F-2. 利用者管理(全スタッフが編集可・ログ記録対象)
- 利用者の一覧 / 登録 / 編集 / 詳細表示
- 保持する属性:
  - 氏名、ふりがな
  - 状態: `利用中 / 休止中 / 入院中 / 終了`
  - 新人同行: `同行可 / 要事前連絡 / 同行不可` の3区分(バッジ表示)
    ※「事前連絡」は新人を同行させる前に利用者へ連絡が必要、という意味の区分
  - スタッフ性別の限定: `なし / 女性のみ / 男性のみ`(バッジ表示)
  - 備考(自由記述)
- 編集は全スタッフ可能。**誰が何を変更したかはすべてログに記録**される(F-5)
- 利用者詳細ページで、その利用者の基本ルート(担当スタッフ・曜日・時間)を一覧表示する
  → 課題6「休止時にどの訪問を止めればいいか探すのが大変」を解決

#### F-3. 基本ルート管理(全スタッフが編集可・ログ記録対象)
- 定期訪問の登録 / 編集 / 削除(削除も論理削除とし、ログに残る)
- 保持する情報: 利用者、担当スタッフ、サービス区分(看護/リハ)、曜日(月〜日)、開始〜終了時刻、頻度
- 頻度は次の3パターンに対応(**デフォルトは「毎週」**):
  | 頻度 | 例 | 入力方法 |
  |------|----|---------|
  | 毎週 | 毎週火曜 | 「毎週」を選ぶだけ(デフォルト選択) |
  | 第n週 | 第2・4週の火曜 | 第1〜5週をチェックボックスで指定 |
  | 2週ごと | 前回訪問から2週間後 | 「2週ごと」を選び、基準日(直近の訪問日)を指定 |
- 編集は全スタッフ可能。**誰が何を変更したかはすべてログに記録**される(F-5)

#### F-4. 変更登録(全スタッフ・即時反映)
- フォームは **選択式中心** で、文章を書かずに登録できる
- 入力項目:
  | 項目 | 形式 |
  |------|------|
  | 利用者 | 選択(検索つきプルダウン) |
  | 対象の基本ルート | 利用者を選ぶと候補が表示される(看護/リハの区別もここで確定) |
  | 種別 | `休み / 振替 / 休止 / 再開` |
  | 対象日 | 日付選択 |
  | 振替先(種別=振替のとき) | 日付・時刻・担当スタッフ |
  | 理由 | 選択(`受診 / 体調不良 / 入院 / 用事 / その他`)+ 自由記述 |
  | ケアマネ連絡状況 | `未連絡 / 連絡済み` の2値 |
- **登録した瞬間に週間スケジュールへ反映される**(承認は不要)
- 登録した変更は編集できる(例: ケアマネ連絡状況を「未連絡」→「連絡済み」に更新)
- 取り消しもできるが、**物理削除はせず「取り消し済み」として記録が残る**

#### F-5. 変更ログと管理者確認(承認に代わる監督機能)
**(a) 変更ログ(全スタッフ閲覧可)**
- 以下の操作がすべて自動で記録される:
  - 変更登録の `登録 / 編集 / 取り消し`
  - **利用者情報の `登録 / 編集`**
  - **基本ルートの `登録 / 編集 / 削除`**
  - スタッフ休暇の `登録 / 削除`
- ログには「誰が・いつ・どのデータの・何を・どう変えたか」が残る(編集は変更前→変更後の差分つき)
- ログは誰にも削除・改変できない

**(b) 管理者の確認チェック(管理者のみ)**
- 各変更に「確認」ボタンがあり、管理者が内容を見てチェックを付ける
- **チェックの有無はスケジュール反映に影響しない**(承認ではない)
- 未確認の変更は、週間スケジュール上に ⚠ マークつきで表示される
- ヘッダーに未確認件数バッジを常時表示 → 見落としを防止
- ケアマネ連絡状況が「未連絡」の変更は一覧上で強調表示する
  → 課題5「ケアマネ連絡漏れによるクレーム」を防止

**(c) 今週の変更一覧パネル**
- 週間スケジュール画面に、その週に適用されている変更を一覧表示する
- `基本ルート(変更前)→ 変更後` を並べて表示し、何がどう変わったか一目で分かる
  - 例: `鈴木一郎様 リハ｜火 10:00-10:40(山田)→ 6/17(水)11:00-11:40(西脇)に振替｜登録: 西脇 6/10 14:32｜CM連絡済 ✓確認済`
- 管理者はこのパネルから直接「確認チェック」を付けられる
  → 「基本ルートからの変更点を見ながらミスがないか確認する」という管理者の作業がここで完結する

#### F-6. 週間スケジュール表示(全スタッフ)
- **看護ページ / リハビリページ** をタブで切り替え
- 表形式: 縦=時間帯、横=スタッフ、紙のスケジュールと同じ見た目を踏襲
- 曜日は **月〜日** の7日間で表示
- 週送りボタン(前週/今週/翌週)で任意の週を表示
- **「自分の予定のみ表示」切替**(スマホでの閲覧を想定。自分の列だけに絞り込める)
- 表示ルール(色分け):
  | 状態 | 表示 |
  |------|------|
  | 通常訪問(基本ルート通り) | 白背景 |
  | 休み | グレー + 打ち消し線 |
  | 振替 | 緑背景(振替先のコマに表示。元コマは打ち消し+矢印表示) |
  | 管理者が未確認の変更 | 上記の色 + 黄色の枠と ⚠ マーク |
  | 休止/入院中の利用者 | 表示しない(利用者詳細に履歴は残る) |
  | スタッフ休暇 | 該当スタッフの列をグレー表示 |
- 変更のあるコマをタップ/クリックすると詳細(理由・登録者・ケアマネ連絡状況・確認状態)が見られる
- 利用者名の横に属性バッジを表示(新人同行不可 / 要事前連絡 / 女性限定 / 男性限定)
  → 課題7を解決。週間スケジュール上でも確認できるようになる

#### F-7. スタッフ休暇登録
- 全日 / 午前半休 / 午後半休 / 時間休 を登録できる
- 一般スタッフは自分の休暇のみ、管理者は全員分を登録できる
- 週間スケジュールに反映される → 課題8を解決

#### F-8. Chatwork通知(カリキュラム要件のAPI活用)
- 以下のタイミングで Chatwork のグループチャットに自動投稿する:
  1. 変更が登録されたとき(即時反映なので、これが全員への周知を兼ねる)
  2. 変更が取り消されたとき
- 投稿フォーマット例:
  ```
  [info][title]📝 スケジュール変更[/title]
  利用者: 田中太郎 様(リハビリ)
  種別: 休み / 対象日: 6/12(金)
  理由: 定期受診
  ケアマネ: 連絡済み
  登録者: 西脇
  [/info]
  ```
- 開発・デモでは **テスト用グループチャット** に投稿する(本番チャットには接続しない)
- Chatwork APIはフリープランでも利用可能(個人アカウントでトークン発行できる)

### 4.2 MVPに含めない機能(第2段階以降の候補)
- カレンダー形式表示(FullCalendar等)
- Google Calendar 連携
- 月間の訪問実績集計(件数・実施率)
- スマホ専用の「今日のマイ予定」リスト表示(MVPは「自分のみ表示」+レスポンシブで対応)
- 看護/リハで管理者権限を分離
- 未確認のままn日経過した変更のリマインド通知

## 5. 非機能要件

| 項目 | 内容 |
|------|------|
| 個人情報 | 実在の利用者・スタッフのデータは一切使用しない。seedデータ(ダミー)で動作させる |
| 認証情報 | Chatwork APIトークン等はコードに直接書かず、環境変数で管理する |
| 監査性 | 変更ログはアプリ上のどの操作でも削除・改変できないこと |
| **対応端末** | **PC(Chrome / Edge)に加え、スマートフォン(iOS Safari / Android Chrome)に対応。訪問スタッフは訪問先からスマホで利用するため、レスポンシブ対応は必須要件とする** |
| テスト | 開発期間が短いため、心臓部である `visit_on?`(3種頻度の判定)の単体テストを最優先で書く。他は手動の動作確認でカバーし、提出後に拡充する |

## 7. 本アプリで解決される課題の対応表

| 課題(1.2参照) | 解決する機能 |
|---|---|
| 1. 転記に時間がかかる | F-4(登録=即反映。転記・承認作業そのものが消滅) |
| 2. 報告様式がバラバラ | F-4(選択式フォーム) |
| 3. PDFが古くなる | F-6(Webでリアルタイム表示。スマホからも確認可) |
| 4. 反映漏れ・報告漏れ | F-4(即時反映で反映漏れが構造的に発生しない)+ F-5(ログと確認チェック) |
| 5. ケアマネ連絡が不明 | F-4 + F-5(連絡状況ステータスと未連絡の強調表示) |
| 6. 休止時のルート検索 | F-2(利用者詳細から基本ルートを逆引き) |
| 7. 同行可否・性別限定 | F-2 + F-6(属性バッジ。同行は可/要事前連絡/不可の3区分) |
| 8. 休みが共有されない | F-7(休暇登録とスケジュール反映) |
| (新)全員編集による誤操作リスク | F-5(消えないログ + 変更一覧パネル + 確認チェック) |
